Three approaches to the draggable sidebar

All three produce the same end result. The difference is implementation strategy.

A

Custom React hook + pointer events (recommended)

A useSidebarResize hook handles drag via onPointerDown/Move/Up, clamps width between 160px–360px, snaps to hidden below 120px, and persists to localStorage. A ResizableShell client component replaces the current .shell CSS grid, rendering the sidebar width as an inline CSS variable.

Pros

  • Zero new dependencies
  • Full control over snap, toggle, animation
  • Fits existing code style exactly

Cons

  • ~80 lines of drag logic to implement carefully
B

react-resizable-panels library

Replace the shell grid with <PanelGroup>, <Panel>, and <PanelResizeHandle> from the react-resizable-panels package. Built-in keyboard resize, touch support, and collapse.

Pros

  • Keyboard accessible out of the box
  • Battle-tested edge cases (touch, a11y)

Cons

  • New dependency (~15 kB)
  • Library panel API differs from current CSS grid — more layout rework
  • Less control over snap / collapse animation
C

CSS resize: horizontal (native browser)

Add resize: horizontal; overflow: auto; to the sidebar. No JS needed — browser handles the drag natively.

Pros

  • ~3 lines of CSS, zero JS

Cons

  • Native handle looks out of place (bottom-right grip, not a center splitter)
  • No snap-to-hidden
  • No toggle button
  • No localStorage persistence
Visual result (all approaches look the same)
